# HomeStream 治理白皮书

> 起心动念皆因果·不靠法律筑墙靠维度拉开距离

HomeStream 采用 **三层分治** 治理架构，在开源普惠与核心壁垒之间建立清晰的边界。
本文档定义了开源边界、许可证策略、依赖审计规则与反 Vibe Coding 机制，
是所有贡献者和使用者的治理参考。

---

## 一、三层分治架构

### 设计哲学

铸钥匠不造墙，只铸钥。三层分治不是"藏私货"，而是按"可教可验"与"需悟"的维度自然分割：

| 层级 | 定位 | 开源策略 | 维度映射 | 类比 |
|:-----|:-----|:---------|:---------|:-----|
| **L1 基础设施层** | 术 — 基础工具与协议 | 完全开源 (MIT) | V1-V3 | 铺路石 |
| **L2 差异化层** | 法 — AI 原生工具链 | 差异化开源 (MIT) | V4-V5 | 路标与驿站 |
| **L3 核心壁垒层** | 道·果 — 维度哲学与商业闭环 | 自用不开源 | V6-V7 | 观星台 |

### L1：基础设施开源

**目标**：让每个人零成本接入 AI 世界。

| 项目 | 说明 | 文件 |
|:-----|:-----|:-----|
| EventStream 引擎 | 事件驱动 pub/sub + 因果链 | `event_stream.py` |
| EventStore 持久化 | SQLite WAL + 事件回放 | `event_store.py` |
| ICP v1.1 协议 | 9种消息类型 + HMAC 防时序攻击 | `event_stream.py` |
| ModelRouter | 硬件自适应三层路由 + 双保障 | `model_router.py` |
| 可观测性中间件 | Prometheus RED 指标 + request_id 追踪 | `middleware.py` |
| 可观测性仪表盘 | 8面板 ECharts 可视化 | `observatory.py` |
| WorktreeManager | Git worktree 并行隔离 | `worktree_manager.py` |
| FastAPI 服务器 | 76 路由完整 REST API | `bridge_v7_server.py` |

### L2：差异化开源

**目标**：提供比竞品更友好的开发体验，但不暴露核心维度。

| 项目 | 说明 | 差异化优势 |
|:-----|:-----|:-----------|
| 灵魂配置系统 | 四层记忆 + 性格参数 | 比 CrewAI 的 YAML 更丰富 |
| 工作流引擎 | 可视化工作流定义 | 比 LangGraph 更轻量 |
| 插件市场骨架 | 签名 + 沙箱 + 注册表 | 竞品无此层 |
| 多平台 IM 网关 | 企微/飞书/钉钉统一接口 | 竞品无此层 |
| 千面设计市场 | 9主题 + 主题标准接口 | 竞品无此层 |
| Skill Router v2 | 双层路由 + SkillsBench | 比 DeerFlow 更完整 |
| 记忆演化引擎 | ReMe 压缩 + 遗忘曲线 | 比 Mem0 更轻量 |
| 间接注入防护 | prompt_security 模块 | 安全维度独有 |

### L3：核心壁垒自用（不开源）

**目标**：以维度拉开距离，不以法律筑墙。

| 项目 | 说明 | 不开源理由 |
|:-----|:-----|:-----------|
| 弹性模式逐级安全 | Solo→Team→Ecosystem 三档 | 商业闭环核心 |
| ICP 自监控 SLA | 协议自健康检查 | 运维壁垒 |
| 正向 AI 培育 | AI 成长轨迹 + 知识果实 | 维度哲学实践 |
| Ratchet Loop 只进不退 | 棘轮锁定 + 双层工坊 | 质量工程核心 |
| 三层免费路由调优 | L1/L2/L3 动态权重 | 运维经验积累 |
| V6-V7 维度哲学 | 开悟框架 + 因果方法论 | 需悟，不可教 |

> **注意**：L3 不开源不是"藏代码"，而是"藏维度"。代码可以被重写，维度无法被复制——
> 一个人对 AI 的理解深度、对生态的长期投入、对因果的敬畏之心，这些才是真正的壁垒。

## 二、许可证策略

### 2.1 主许可证：MIT

HomeStream 开源代码统一采用 **MIT License**。

**选择理由**：
- 最低门槛：任何人可商用、修改、分发，无需开源衍生品
- 最大兼容：与 Apache 2.0、BSD、MPL 等主流协议兼容
- 符合定位：托底普大众化接入 AI 世界，不应被协议条款阻碍

### 2.2 贡献政策：不要求 CLA

HomeStream **不使用 CLA（Contributor License Agreement）**。

**理由**：
- CLA 提高参与门槛，与"分享型开源"定位冲突
- MIT 协议已足够保护项目（贡献者自动授予 MIT 权利）
- 不签合同，只递钥匙——钥匙到了你手上，星光之旅，就此起航

**贡献者须知**：
- 提交 PR 即表示同意以 MIT 协议开源你的贡献
- 无需签署任何额外文件
- 贡献者保留版权，项目使用 MIT 许可

### 2.3 协议变更禁令

> **规则**：MIT 协议一旦发布，不可变更为更严格的协议（如 GPL、AGPL）。

**理由**：
- 协议变更是对早期贡献者和使用者的违约
- MIT → GPL 会迫使所有衍生品被迫开源，破坏信任
- 如果需要保护新功能，新功能以独立项目 + 独立协议发布，不追溯变更现有代码

## 三、依赖审计规则

### 3.1 GPL/AGPL 禁令

> **规则**：L1/L2 层代码 **禁止引入 GPL、AGPL、SSPL 许可证的依赖**。

**理由**：
- GPL 系列"传染性"会迫使 HomeStream 整体被迫转为 GPL
- AGPL 更危险：网络使用也触发开源义务
- SSPL 是 MongoDB 发明的"假开源"协议，实质是商业许可

**允许的协议**（白名单）：
| 协议 | 类型 | 典型项目 |
|:-----|:-----|:---------|
| MIT | 宽松 | pytest, requests |
| Apache 2.0 | 宽松 | FastAPI, pydantic |
| BSD 2/3-Clause | 宽松 | structlog |
| MPL 2.0 | 弱传染（文件级） | — |
| Python Software Foundation | 宽松 | Python stdlib |
| Unlicense / CC0 | 公共领域 | — |

### 3.2 审计流程

```bash
# 安装审计工具
pip install pip-licenses

# 检查所有依赖的许可证
pip-licenses --format=table --with-license-file

# 只显示 GPL/AGPL 风险
pip-licenses --fail-on="GPL;AGPL;SSPL" --summary
```

**CI 集成**（`.github/workflows/license-audit.yml`）：
```yaml
- name: License audit
  run: |
    pip install pip-licenses
    pip-licenses --fail-on="GPL;AGPL;SSPL" --summary
```

### 3.3 L3 例外

L3 核心壁垒层代码不对外发布，可使用任何协议的依赖。
但仍建议优先选择宽松协议，便于未来可能的 L3→L2 降级开源。

## 四、反 Vibe Coding 机制

### 4.1 什么是 Vibe Coding

Vibe Coding 指仅凭 AI 生成代码、不加审查就合并的行为。它会：
- 引入未审计的安全漏洞
- 产生幻觉API调用（调用不存在的库函数）
- 破坏架构一致性（每个PR一个风格）
- 最终杀死开源项目的信任基础

### 4.2 HomeStream 的六道防线

| 防线 | 机制 | 实现位置 |
|:-----|:-----|:---------|
| **1. 人工审查** | 所有 PR 必须至少一名 maintainer 审查 | CONTRIBUTING.md |
| **2. CI 强制门禁** | 测试 + 覆盖率 + lint + 安全扫描必须全绿 | `.github/workflows/` |
| **3. 覆盖率底线** | 不低于 70%，新增代码必须配套测试 | `pyproject.toml` |
| **4. 安全扫描** | bandit 安全扫描 + pip-audit 依赖漏洞扫描 | `.pre-commit-config.yaml` |
| **5. 架构守护** | 核心模块有类型标注 + 接口契约 + 测试断言 | `providers/base_provider.py` |
| **6. Ratchet Loop** | 实验代码必须通过 Maker→Reviewer 双层工坊才合并 | `ratchet_loop.py` |

### 4.3 AI 辅助编码规范

HomeStream 欢迎 AI 辅助编码，但要求：

1. **AI 生成的代码必须标注**：commit message 中注明 `AI-assisted`
2. **必须人工逐行审查**：不能"看起来对就合并"
3. **必须通过全部测试**：AI 生成的代码必须配套测试
4. **不得引入未审计依赖**：AI 推荐的 pip install 必须经过许可证审计
5. **不得绕过架构约束**：AI 不得修改 `base_provider.py` 等接口契约文件

## 五、商标与知识产权

### 5.1 商标策略

> **规则**：商标必须注册，代码不申请专利。

| 商标 | 注册类别 | 状态 |
|:-----|:---------|:-----|
| HomeStream | 第9类（软件）、第42类（SaaS）、第38类（通讯） | 规划中 |
| 铸钥匠 | 第9类、第42类 | 规划中 |
| OpenBridge | 第9类、第42类 | 规划中 |

**理由**：
- 商标保护品牌识别度，防止他人冒用 HomeStream 名义分发恶意软件
- 不申请专利：专利要求公开技术细节，且会限制他人使用，与开源精神冲突
- 每年评估商标维护状态

### 5.2 版权归属

- 代码版权归贡献者各自所有
- 项目以 MIT 协议统一授权使用
- 不设 CLA，不集中版权到单一实体

### 5.3 知识书阁物理隔离

九重书阁（知识库）与 HomeStream 开源代码 **物理隔离**：
- 书阁数据不进入 Git 仓库
- 书阁服务端口（3460）与开源版端口（3458）独立
- `.openbridge_mode` 文件控制运行模式：`opensource` 模式下书阁不可达

## 六、分享型开源定位

### 6.1 不是商业型开源

HomeStream 是 **分享型开源**，不是商业型开源。

| 维度 | 商业型开源 | 分享型开源 (HomeStream) |
|:-----|:----------|:----------------------|
| 目标 | 通过开源获客，最终卖企业版 | 通过开源分享，让更多人接入 AI |
| 核心代码 | 开源阉割版 + 商业完整版 | 开源完整可用版 + 自用维度版 |
| 盈利模式 | 订阅/许可/咨询 | 不以盈利为首要目标 |
| 社区定位 | 用户 → 客户转化 | 同行者 → 生态共建 |
| 协议策略 | 双协议（开源+商业） | 单一 MIT |

### 6.2 商业化路径（可选，非必须）

HomeStream 的商业化是 **自然溢出**，不是刻意设计：

| 路径 | 说明 | 门槛 |
|:-----|:-----|:-----|
| 托管服务 | 提供云端 HomeStream 实例 | 需要服务器成本 |
| 定制开发 | 为企业定制 Agent 工作流 | 需要人力投入 |
| 培训咨询 | AI 架构运用分享与培训 | 需要内容积累 |
| 主题市场 | 千面设计市场优质主题付费 | 需要社区规模 |

> **原则**：商业化是分享的自然结果，不是分享的目的。
> 如果商业化欲望盖过了分享初心，项目就变味了。

## 七、开源 ≠ 免费

### 7.1 开源的含义

开源（Open Source）意味着：
- **源代码公开可审**：任何人可以查看、审计代码
- **可自由使用**：商用、修改、分发无需额外许可
- **可自由贡献**：任何人可以提交 PR

### 7.2 开源不意味着

- **不意味着免费支持**：issue 回复、bug 修复不保证时效
- **不意味着免费托管**：自行部署，或付费使用托管服务
- **不意味着免费咨询**：架构咨询、定制开发需要付费
- **不意味着放弃商标**：HomeStream 名称和 logo 受商标保护

### 7.3 合理使用边界

| 行为 | 是否允许 |
|:-----|:---------|
| 个人学习使用 | 允许 |
| 公司内部使用 | 允许 |
| 商业产品集成 | 允许（MIT） |
| 分发修改版 | 允许（保留版权声明） |
| 冒用 HomeStream 商标 | **禁止** |
| 移除版权声明 | **禁止** |
| 引入 GPL 依赖污染 | **禁止** |

- [ ] 无 GPL/AGPL/SSPL 依赖引入
