# HomeStream 技术主权保障评估报告

> **定位**：开源前技术"卡脖"风险全景评估 + 保障方案
> **日期**：2026-07-09
> **触发**：九重担忧"架构模式在技术层面被卡脖，涉及运维网络专业知识"
> **结论**：HomeStream 架构天然具备"本地优先"抗卡脖基因，但有 **1项P0致命缺口** 需立即修复

---

## 一、执行摘要

九重的担忧切中要害。开源不是"代码放出去"就完了——如果技术底座被人"卡脖"，开源就变成了"送人一把打不开的锁"。

经过对 HomeStream 全部技术依赖链的逐层盘点，我们发现：

| 维度 | 评估 |
|:-----|:-----|
| **架构抗卡脖能力** | ★★★★☆ 优秀——三层路由+双保障+本地优先，天然抗断供 |
| **当前P0致命缺口** | **1项**——jsDelivr CDN 在中国IPv4已不可访问，可观测性面板受影响 |
| **P1高危风险** | 3项——模型下载/推理引擎安装/API单点依赖 |
| **P2中危风险** | 2项——Python包安装/代码托管 |
| **无需担忧项** | 数据库/本地服务/协议层——全部自主可控 |

**核心判断**：HomeStream 的架构哲学（L1零配置本地优先 + L3 Ollama托底）本身就是最强的抗卡脖设计。只要补齐1个P0缺口 + 完善3项P1保障，就能实现"任何环节被卡，系统仍然可用"的技术主权目标。

## 四、各风险点详细分析

### 4.1 [P0] jsDelivr CDN — 可观测性面板的致命单点

**问题定位**：
```python
# bridge_v7_server.py 中的硬编码CDN引用
<script src="https://cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js"></script>
```

**风险分析**：
- jsDelivr因ICP备案问题，在中国IPv4网络下**已无法访问**（2022年至今）
- 仅IPv6可达，但中国大部分家庭/办公网络仍以IPv4为主
- 直接后果：用户打开可观测性仪表盘 → 白屏，ECharts不加载 → 8个监控面板全部不可用
- 这是当前**唯一影响功能正常使用的P0级卡脖点**

**保障方案**：
```
方案A（推荐）：ECharts 本地化自托管
  ├── 将 echarts.min.js 下载到 assets/ 目录
  ├── 修改引用路径为 /assets/echarts.min.js
  ├── 添加静态文件路由（与PWA图标路由同机制）
  └── 优势：完全离线可用，零外部依赖

方案B（备选）：多CDN容灾
  ├── 主CDN: cdn.bootcdn.net（国内CDN，已备案）
  ├── 备CDN: cdn.staticfile.org（国内CDN）
  ├── 降级: 本地文件
  └── 劣势：仍依赖外部CDN，不如方案A彻底
```

### 4.2 [P1] HuggingFace 模型下载 — 安装阶段卡脖

**问题定位**：
- L1层推荐使用Qwen系列模型（`qwen_model.gguf`），权重托管在HuggingFace
- GFW在TCP层阻断 `huggingface.co`，下载需配置 `HF_ENDPOINT=https://hf-mirror.com`

**现有缓解**：
- `hf-mirror.com` 镜像（社区维护，非官方）
- 文档中已提及镜像配置方法

**残留风险**：
- `hf-mirror.com` 是个人/社区维护的镜像，存在关停风险
- 镜像带宽有限，大模型下载可能超时

**保障方案**：
```
双镜像策略：
  ├── 主路径: ModelScope (魔搭) ← 阿里达摩院官方平台，国内服务器
  │   ├── 服务器在国内，无需代理
  │   ├── Qwen系列官方同步（阿里自家模型）
  │   ├── pip install modelscope → model download
  │   └── 风险：极低（阿里基础设施保障）
  │
  ├── 备路径: hf-mirror.com ← 社区镜像
  │   └── 环境变量 HF_ENDPOINT=https://hf-mirror.com
  │
  └── 兜底: GitHub Release 附件
      ├── 将推荐模型(.gguf)打包为Release附件
      └── 用户直接从GitHub/Gitee下载
```

### 4.3 [P1] Ollama 安装下载 — 推理引擎卡脖

**问题定位**：
- Ollama安装包 `ollama.com` 被GFW阻断
- 需通过 `gh-proxy` 等镜像下载

**现有缓解**：
- 文档中提及gh-proxy镜像方案

**保障方案**：
```
三路安装保障：
  ├── 路径1: llama.cpp 替代（推荐）
  │   ├── llama.cpp从GitHub源码编译，或下载预编译二进制
  │   ├── 无需Ollama，直接用llama-server
  │   ├── HomeStream的L1层原生支持llama.cpp API
  │   └── 优势：更轻量，无额外运行时依赖
  │
  ├── 路径2: gh-proxy镜像安装Ollama
  │   └── https://gh-proxy.com/https://github.com/ollama/ollama/releases/...
  │
  └── 路径3: Gitee镜像Release
      ├── 在Gitee仓库Release中附带Ollama安装包
      └── 用户从Gitee直接下载
```

### 4.4 [P1] 模型API单点依赖 — 运行时卡脖

**问题定位**：
- L2层：GLM API (`open.bigmodel.cn`) — 智谱AI，国内公司
- L3层：DeepSeek API (`api.deepseek.com`) — 深度求索，国内公司
- 虽然都是国内公司，但仍是单一服务商依赖

**现有缓解**：
- 双保障机制：主线路(L1本地+L2 GLM) → 复线(L3 DeepSeek) → L1本地托底
- L1层完全本地（llama.cpp），即使所有API都停服，系统仍可用

**保障方案**：
```
Provider注册表已开放扩展，增加更多国产API备选：
  ├── 已有: GLM(智谱) + DeepSeek(深度求索)
  ├── 可扩展: 通义千问(阿里DashScope) ← API兼容OpenAI格式
  ├── 可扩展: 文心一言(百度千帆) ← API兼容
  ├── 可扩展: Kimi(月之暗面) ← API兼容
  └── 可扩展: 用户自有API(通过ProviderRegistry.register())

文档增加"如何接入新Provider"教程，降低用户自主扩展门槛
```

### 4.5 [P2] PyPI 包安装 — 已有成熟方案

**问题定位**：
- 8个核心依赖（pydantic/fastapi/uvicorn/structlog/prometheus-client/requests/pytest/httpx）
- 全部来自PyPI，中国访问不稳定

**现有缓解**：
- 清华大学TUNA镜像 `pip install -i https://pypi.tuna.tsinghua.edu.cn/simple`

**保障方案**：
```
文档层保障：
  ├── README增加"国内安装指南"章节
  ├── 提供 requirements.txt + 清华镜像一键安装命令
  └── 提供离线安装方案：pip download → 拷贝 → pip install --no-index

代码层保障：
  ├── 所有依赖均为成熟开源项目(MIT/BSD许可证)
  ├── 依赖列表精简（仅8个核心包），无重型依赖链
  └── 可考虑vendor关键依赖（将pydantic等核心库内联到项目）
```

### 4.6 [P2] GitHub 代码托管 — 已有双推计划

**问题定位**：
- 代码托管在GitHub，存在封号/地区限制风险
- 2026年已有OpenClaw等AI项目被GitHub封禁的先例

**现有缓解**：
- Gitee镜像双推送（7/10计划执行）

**保障方案**：
```
双源双托管：
  ├── 主源: GitHub (github.com/Ninefoldatwill/OpenBridge)
  ├── 镜像: Gitee (gitee.com/...)
  ├── README中双源链接并列
  └── Release同时发布到两个平台

长期保障：
  ├── 考虑自建Gitea实例（完全自主可控）
  └── 关键Release版本打包到网盘/对象存储
```

## 五、架构级抗卡脖能力评估

### 5.1 协议层 — 无卡脖风险 ✅

| 协议 | 来源 | 许可证 | 风险评估 |
|:-----|:-----|:-------|:---------|
| ICP v1.1 | HomeStream自有 | MIT | 零风险，自主定义 |
| A2A | Google开源 | Apache 2.0 | 低风险，开源协议不受单一实体控制 |
| MCP | Anthropic开源 | MIT | 低风险，已提交开放标准 |
| OpenAI兼容API | 行业事实标准 | 无限制 | 零风险，API格式无版权约束 |

**结论**：协议层全部基于开源许可证，不存在"协议被收回"的风险。即使Google/Anthropic退出协议维护，已发布的版本仍然有效，社区可以fork继续维护。

### 5.2 框架层 — 低卡脖风险 ✅

| 框架 | 许可证 | 依赖深度 | 风险评估 |
|:-----|:-------|:---------|:---------|
| Python | PSF License | 运行时 | 零风险，开源+国内镜像 |
| FastAPI | MIT | HTTP服务 | 低风险，轻量可替代(标准库http.server) |
| Uvicorn | BSD-3 | ASGI服务器 | 低风险，可替代(granian/hypercorn) |
| Pydantic | MIT | 数据校验 | 低风险，可手写校验替代 |
| SQLite | Public Domain | 数据库 | 零风险，公有领域无版权 |
| structlog | MIT | 日志 | 低风险，可替代(标准库logging) |
| requests | Apache 2.0 | HTTP客户端 | 低风险，可替代(urllib) |

**结论**：所有框架依赖均为成熟开源项目，许可证宽松。即使极端情况下某个库停止维护，代码已开源，社区可以fork。且HomeStream的依赖列表极简（仅8个核心包），替代成本低。

### 5.3 模型层 — 中等卡脖风险 ⚠️

| 模型 | 来源 | 许可证 | 下载渠道 | 风险评估 |
|:-----|:-----|:-------|:---------|:---------|
| Qwen系列 | 阿里通义 | Apache 2.0 / Qianwen License | HuggingFace + ModelScope | 中风险（HF受限，ModelScope可用） |
| DeepSeek | 深度求索 | MIT | API + HuggingFace | 中风险（API单点，但可本地部署） |
| GLM | 智谱AI | MIT | API + HuggingFace | 中风险（同上） |

**结论**：模型权重均以宽松许可证开源，不存在"模型被收回"的法律风险。但下载渠道受限是实际问题，ModelScope(魔搭)是最可靠的国产替代路径。

### 5.4 运维网络层 — 需关注但不构成卡脖 ⚠️

九重特别提到的"运维网络专业知识"担忧，逐项分析：

| 运维项 | 风险 | HomeStream现状 | 保障措施 |
|:-------|:-----|:---------------|:---------|
| **DNS解析** | 域名劫持/污染 | 使用阿里223.5.5.5 | 配置文档提供多DNS备选 |
| **端口绑定** | 端口冲突/被占用 | localhost:3458等固定端口 | 文档说明如何修改端口 |
| **防火墙** | 系统防火墙拦截 | 本地通信为主 | 文档提供防火墙配置指引 |
| **进程守护** | 服务意外退出 | nssm服务(Windows) | 文档提供systemd/launchd方案 |
| **反向代理** | Nginx配置复杂 | 可选(非必须) | 文档提供Nginx示例配置 |
| **SSL证书** | HTTPS配置复杂 | 可选(非必须) | 文档提供Let's Encrypt方案 |
| **公网部署** | 安全暴露风险 | 默认仅本地 | 文档强调安全边界+提供内网穿透方案 |

**结论**：运维网络问题不构成"卡脖"——HomeStream默认运行在localhost，用户无需公网IP、域名、SSL证书即可使用。运维复杂度仅在"可选的公网部署"场景出现，且有成熟文档指引。

## 六、技术保障方案总览

### 6.1 已实现的抗卡脖机制

| 机制 | 实现 | 抗卡脖效果 |
|:-----|:-----|:-----------|
| **L1零配置本地优先** | llama.cpp本地推理，无需API Key | 即使所有外部API停服，系统仍可用 |
| **三层路由逐级降级** | L1→L2→L3自动降级 | 单层故障不致命 |
| **双保障自动切换** | 主线路超时→复线DeepSeek | 网络波动自动容灾 |
| **SQLite本地存储** | 无外部数据库依赖 | 数据主权完全自主 |
| **Provider注册表开放** | 第三方可注册新Provider | 不受限于预置API |
| **ICP自有协议** | 不依赖外部协议标准 | 协议演进自主可控 |
| **纯Python标准库优先** | urllib而非aiohttp/httpx(核心路径) | 减少第三方依赖 |

**场景3：GitHub 账号被封**
