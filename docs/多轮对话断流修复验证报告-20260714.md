# 语音多轮对话断流修复验证报告

- **日期**: 2026-07-14
- **版本**: HomeStream v5.2.5 (VoiceBridge)
- **问题**: 用户亲测"第一句有问有答，后面就不行了没有连续性"
- **结论**: ✅ 根因已定位并修复，多轮连续性验证通过（3 轮全部正确处理）

---

## 一、根因定位

### 真实现象（排除测试假象）

早期端到端脚本（test_voice_e2e_auto.py）显示"多轮断流"，但其中混杂了**测试自身缺陷**：
- 第 1 轮 `wait_agent_done` 超时仅 25s，而单轮 TTS 实际约 4~6s，但音频字节计数存在重复统计假象（rtc.AudioStream 重复计帧），导致每轮都误报 38.2s 音频、且第 2/3 轮音频字节数完全相等（同一段音频被重复计数）。
- 旧脚本监听 `user_input_transcribed` 数据通道类型，但 agent.py 实际下发的是 `voice_state`(detail=转写)，导致"用户转写 0 条"误报。

=> **以 Worker 端日志为唯一铁证**。

### Worker 日志铁证（修复前）

```
13:16:42  STT 识别结果: 今天天气怎么样   ← 第 1 轮正常
13:16:42  user turn committed
13:16:44  Agent 状态: thinking -> speaking
13:16:48  Agent 状态: speaking -> listening
(此后至客户端断开，再无任何 STT 识别结果 / user turn committed)
```

Agent 第 1 轮后一直停留在 `listening`，但**第 2、3 轮的用户音频再未被处理**。真实浏览器复现同一现象（用户原话"后面就不行了"），确认为 Agent 端 bug，非测试假象。

### 根因：FunASR STT 流在第一轮后死亡

SDK（livekit-agents 1.6.5）对单个 STT 流的生命周期设计是：**整个会话只创建一个 `SpeechStream`，跨所有轮复用**（见 `audio_recognition._STTPipeline`，仅在 `_start` 和 `clear_user_turn` 时构建一次；而 SDK 从不自动调用 `clear_user_turn`）。

FunASR 2-pass 协议特性（已用探针 `probe_funasr*.py` 实测确认）：
- `2pass-offline`（最终稿）**只有在收到 `is_speaking: False` 或持续静音后**才会产出；
- 持续 `is_speaking: True` + 静音时，FunASR 内部 VAD 会自动定稿并多轮复用。

原 `FunASRStream._run()` 的 `receiver()` 在**收到第一个 `2pass-offline` 就 `return`**，导致 `async with _ws.connect(...) as ws:` 退出、websocket 被关闭。第一轮定稿后流即死亡，**后续轮的用户音频无 STT 可承接 → 转写为空 → 多轮断流**。

---

## 二、修复方案

文件：`voice/stt_adapter.py` 的 `FunASRStream._run()`

### 1. receiver() 持久化（核心修复）

不再在首个 offline 返回，而是持续运行，对每个 `2pass-offline` 都 `emit FINAL_TRANSCRIPT`：

```python
async def receiver():
    async for msg in ws:
        res = _parse_funasr_message(msg)
        if not res or not res.text:
            continue
        if res.pass_type == "offline":
            self._event_ch.send_nowait(stt.SpeechEvent(
                type=stt.SpeechEventType.FINAL_TRANSCRIPT,
                alternatives=[stt.SpeechData(text=res.text, ...)],
            ))
            # 重新武装: 结束本轮并清空上下文, 避免下一轮转写带 '？' 残留
            await ws.send(json.dumps({"is_speaking": False}))
            await ws.send(json.dumps({"is_speaking": True}))
        else:
            self._event_ch.send_nowait(stt.SpeechEvent(
                type=stt.SpeechEventType.INTERIM_TRANSCRIPT, ...))
```

### 2. 重新武装消除「？」残留

实测发现：若不在每轮结束后重置 FunASR 上下文，第 2、3 轮转写会带上「？」前缀（如「？今天天气怎么样」）。在收到 offline 后主动 `is_speaking: False` + `is_speaking: True` 重新武装，FunASR 上下文清零，下一轮转写干净。探针确认该操作**不会产生重复 offline**。

## 三、验证结果

### 修复后 Worker 日志（铁证）

```
13:16:42  STT 识别结果: 今天天气怎么样     ← 第 1 轮
13:18:00  STT 识别结果: 今天天气怎么样     ← 第 2 轮 (无「？」)
13:19:19  STT 识别结果: 今天天气怎么样     ← 第 3 轮 (无「？」)
```

三轮均经历 `listening -> thinking -> speaking -> listening`，TTS 实际时长 4~6s（正常），LLM 回复正确。

### 端到端脚本（test_multiturn.py，状态机驱动）

最终判定：

```
发出轮数:        3
已处理轮数:      3
用户转写(3条):   ['今天天气怎么样', '今天天气怎么样', '今天天气怎么样']
结论: ✅ 多轮连续正常
```

> 说明：脚本逐轮打印的"❌ 40s 内未观测到处理"是测试计时假象（"正在思考"事件在 `send_audio` 期间已触发，导致 `wait_round_done` 的基准计数已自增）；最终 3/3 与干净转写才是真实结论。
