## 二、交互语音开发模块冲浪

### 2.1 六强框架选型地图（2026）

| 框架 | 定位 | 架构范式 | 首包延迟 | 开源/托管 | 适合场景 | 对 HomeStream 启示 |
|------|------|----------|----------|-----------|----------|-------------------|
| **LiveKit Agents** | 开源语音 Agent 框架 + 平台 | 级联（VAD+STT+LLM+TTS 可调） | 500–800ms | Apache-2.0 可自托管 | 多租户、WebRTC+电话、需要可控 | ✅ 咱们已选，路线正确 |
| **Pipecat** | 开源流水线框架 | FrameProcessor 管道 | <800ms | BSD-2-Clause | Python 优先、想完全自己编排 | 管道抽象最干净，可借鉴 |
| **Vapi** | 托管平台 | 托管级联 | <1s | 托管（不可自托管） | 最快做出可打电话的 Agent | 不适合咱们"自托底"基因 |
| **Retell** | 托管平台 | 托管级联 | 最低延迟档 | 托管 | 呼叫中心、warm transfer | 同上，闭源 |
| **OpenAI Realtime API** | 原生语音到语音 | 单 provider 闭环 | <300ms | 仅托管 | 延迟至上、可接受锁定 | 可作 L1 语音大模型候选 |
| **Daily Bots** | Pipecat + Daily.co | Pipecat 运行时 | <800ms | 托管控制面 | 已有 Daily.co 用量 | — |

**关键判断**：LiveKit 1.6.5 同时支持级联与 OpenAI Realtime / Gemini Live 作为 drop-in 替代，这种"灵活度"是咱们多租户产品的核心原因。咱们的选型没走偏。

### 2.2 端到端语音大模型浪潮（2026）

| 模型 | 首包延迟 | 开源 | 中文 | 对咱们启示 |
|------|----------|------|------|-----------|
| **Qwen3-Omni** (30B-A3B) | **211ms** 音频 / 507ms 音视频 | ✅ Apache-2.0 | ✅ 强（Wenetspeech WER 4.69% vs GPT-4o 15.3%） | 🔑 **L1 语音大模型自托底最强候选**，可替代"Ollama 文本LLM + EdgeTTS 两段式" |
| **GPT-4o Realtime** | 0.5–1.0s | ❌ 仅 API | 中 | 闭源，仅作对比基线 |
| **Gemini Live** | 0.68s | ❌ 仅 API | 中 | 同上 |
| **Moshi** (开源) | 0.261s | ✅ | 中英追赶中 | 端到端语音 LLM 早期标杆 |
| **Mini-Omni / GLM-Omni** | 1.0s+ | ✅ | ✅ | 开源全模态，可作研究参考 |
| **Freeze Omni** | 1.18s | ✅ 本地 | — | 资源高效但性能低 |

**核心洞察**：开源语音大模型（尤其 Qwen3-Omni）中文已反超闭源，且 211ms 延迟逼近人类反应（250ms）。咱们"铸钥匠免费托底"的终极形态，应是 **Qwen3-Omni 本地化** 而非两段式 ASR→LLM→TTS。

### 2.3 VAD / 打断 / 低延迟 最佳实践（明天可抄）

**① 双层判定（2026 主流做法）**
- VAD 只回答"有没有声音"，回答不了"这声音算不算打断"
- 上层叠 **语义化 turn detection**：区分真打断 / 附和（"嗯""好"） / 噪声
- LiveKit adaptive interruption、Pipecat Smart Turn、Deepgram Flux、VideoSDK Namo 都走模型路线
- **取舍**：安静耳麦场景朴素 VAD 够用；噪音大 / 附和多场景，几十~200ms 模型延迟换误打断下降很值

**② 双 pass VAD（砍延迟到 110–140ms 的 6 lever 之一）**
- WebRTC GMM 快触发（15–20ms）→ 立即 halt TTS
- Silero 确认（150–300ms）→ 若拒绝则恢复播放
- Silero 默认 smoothing 会引入数百 ms 确认滞后，单用会超 150ms

**③ 6 个 VAD 调优杠杆（SyncSoft 2026）**
1. jitter buffer 100ms→40ms（健康网络）
2. 双 pass VAD（GMM 触发 + Silero 确认）
3. 语义 endpointing（用 STT partial 替代帧能量端点）
4. 流式 TTS + 可 flush 播放缓冲（200–300ms，<20ms 清空）
5. VAD+STT+TTS 同 VPC 同 AZ 部署（跨区 RTT 可加 220–260ms）
6. barge-in cooldown 250ms（防 TTS 尾音回声自触发）

**④ 被反复强调的底座前提**
- **AEC 回声消除是第一位的**：无 AEC → AI 自己声音被麦收回 → 无限自打断（dev.to 生产架构蓝图）
- **prefix_padding_ms（200ms）**：捕获句首轻辅音，避免截断首字
- **阈值参考**：`probability_threshold: 0.65, min_speech_duration_ms: 150, min_silence_duration_ms: 700, prefix_padding_ms: 200`
- **状态机 race condition**：server 是最终仲裁者，client truncate 需 server ack，否则 AI 说话中收到"已 idle"会崩溃或丢音频

**⑤ 咱们当前状态对照**
- ✅ 已做：VAD 阈值调到 0.65、min_speech 0.5s、min_silence 1.0s、prefix_padding 0.5s、UI 状态反馈、浏览器 3A 约束
- ⚠️ 待做：双 pass VAD、语义 turn detection、流式 TTS flush、同机部署（咱们已同机，这点是优势）

### 2.4 语音原生可观测性（语音 Agent 的"第三失败面"）

传统 APM 只看 HTTP，看不到语音三失败面：**音频质量（jitter/丢包/编解码）**、**轮次（barge-in 失败/迟端点）**、**ASR 漂移（WER 微升悄悄污染 LLM 上下文）**。

**4 层评估框架（supermia.ai）**：基础设施 / Agent 执行 / 用户体验 / 业务结果
**生产阈值**：WER<5%（clean）/ TTFA<800ms（超 1.5s 用户挂断）/ 容器率>75%
**关键指标**（qubittool 语音 Trace）：
- `time_to_first_audio` 体感响应速度
- `vad_false_commit_rate` 抢话/过早提交比例
- `asr_word_error_rate` 转写准确率
- `tts_underrun_count` 播放是否平滑
- `interruption_rate` 对话自然度
- `tool_latency_p95` 后端瓶颈
- `user_reprompt_rate` 用户不满意程度

**工具**：FutureAGI / Hamming（商业、语音原生）、Langfuse / Helicone（文本层、便宜）、自建 Trace（开源可覆盖仿真+CI 回归）

**对咱们**：明天实测时可加一个轻量"语音链路时间线 Trace"面板（vad→stt→llm→tts→playback 各段耗时），这正是 v5.3.5 可观测性面板的语音子集。

### 3.4 安全与私有化（铸钥匠基因强相关）

**技能市场比浏览器插件更危险**（ClawHub 安全分析）：
- 恶意 Skill 可 prompt injection、数据外泄、执行危险命令、访问凭证、提权
- **Agensi 8 点安全检查**：① prompt injection ② data exfiltration ③ secret detection ④ dangerous commands ⑤ obfuscation ⑥ external fetches ⑦ credential access ⑧ privilege escalation

**私有化范式**（与"免费托底 + 自托底"基因一致）：
- SkillHub：企业内"技能应用商店"，只对公司内部开放，上架经过审核的技能包
- OoderAgent：零部署私有能力库，MIT 开源，P2P 能力分享
- **结论**：铸钥匠的技能/设计市场必须支持 **私有化自托管 + 安全扫描上架门槛**，否则与"数据自主权"初心冲突

## 四、融优结论 & 明日实战路线图
