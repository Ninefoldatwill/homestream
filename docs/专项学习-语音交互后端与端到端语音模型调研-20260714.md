# 专项学习 · 语音交互后端与端到端语音模型调研（2026-07-14）

> 目的：为 HomeStream v5.2.5 语音链路「增强 + 端到端测试」提供技术选型依据。
> 方法：先上网调研（框架对比 / 架构两派 / 端到端语音模型 / 流式 TTS / 可观测性），再结合项目现状给出增强路线。
> 合规：融理念与模式，不复制源码（九重 7/11 再三强调）。

## 一、WorkBuddy / HomeStream 自身语音技能盘点

### 1.1 WorkBuddy 技能市场现况
- SkillHub 语义检索 `voice / TTS / STT / 语音交互` → 命中均为通用 skill（self-improving-agent、find-skills、summarize），相关度 score < 0.05，**无现成「语音交互 / 语音测试」专用 skill**。
- 本地已装 skills 中仅有 `openai-whisper-api`（ASR 转写，可作语音输入补充），无 TTS / 语音对话 / 打断测试类 skill。
- **结论**：测试语音交互不依赖外部 skill，靠 HomeStream 自研链路即可。

### 1.2 HomeStream v5.2.5 已有语音能力（澜舟自己造的）
| 能力 | 现状 | 备注 |
|---|---|---|
| STT | FunASR（自托管 :10096） | 中文优先 |
| LLM | Ollama qwen2.5:3b（本地 :11434） | 已修 LLM 超时子类 `_TimeoutLLM`（强注 60s 渡冷启动） |
| TTS | EdgeTTS（兜底） | 非自托管、音色固定 |
| 传输 | 自托管 LiveKit（:7880） | AgentSession + Worker（agent_name=homestream-voice） |
| VAD | Silero（threshold 0.65 / min_speech 0.5s / min_silence 1.0s） | 已调优，解决「断断续续」 |
| 轮次/打断 | turn_detection="vad" + interruption（vad 模式） | 自托管避 cloud 401 |
| 状态同步 | DataChannel 推 agent/user 状态 | 浏览器「聆听/思考/说话」视觉反馈 |
| 触发 | `/dispatch` 显式派发 Agent | 解决空 agent_name 不可靠 |

**结论**：闭环已验证（initializing→listening→thinking→speaking 干净出声）。待增强点：TTS 自然度/自托管、端到端低延迟、可观测性。

## 二、语音交互后端框架：三层定位（关键认知）

> 核心洞察：**这三个不是同一层的竞品**——选错层 = 三个月后重写。差异化的硬骨头不再是 STT/LLM/TTS，而是 **turn detection（轮次检测/打断）**。

| 工具 | 所处层 | 你拥有什么 | 开源 | 自托管 | 最佳场景 |
|---|---|---|---|---|---|
| **Vapi** | 托管平台 | 什么都不用管 | 否 | 否 | 一周内上电话 Agent、telephony-first |
| **LiveKit Agents** | 实时传输(SFU) + Agents SDK | 基础设施（跑媒体服务器） | Apache-2.0 | ✅ 全链路 | WebRTC-native、多Participant、voice+人工接管 |
| **Pipecat** | 传输无关的 pipeline | 整个 loop（自带 transport） | BSD-2 | ✅ | Python 团队、frame 级控制、研究/多模态实验 |
| **Retell** | 托管（呼叫中心向） | 托管 | 否 | 否 | 企业呼叫中心、warm transfer、合规审计 |
| **Daily Bots** | Pipecat + Daily.co 传输 | OSS 框架 + 托管基建 | BSD-2 | 部分 | 想要 OSS 又不想管基建 |

- 栈自底向上：**transport（音频传输） → orchestration（STT→LLM→TTS loop + 谁在说话） → 应用层**。
- 我们的选择 **LiveKit + 自托管** 完全吻合「免费托底 / 自托管 / 维度竞争」品牌哲学。

## 三、架构两派：级联(cascading) vs 端到端(realtime)

| 维度 | 级联 ASR→LLM→TTS（LiveKit/Pipecat） | 端到端（OpenAI Realtime / Gemini Live / Qwen3-Omni） |
|---|---|---|
| TTFA（首包延迟） | 500–800ms（调优后） | <300ms（部分 <211ms） |
| 灵活性 | 单配置换 provider、细粒度成本/工具调用 | 锁定单一厂商、单一音色族 |
| 语音质量 | 依赖 TTS 选型 | 原生优秀 |
| 计费 | 可预测（按组件） | 按秒 token，难预测 |
| 自托管 | ✅ 完全 | ⚠️ 仅 Qwen3-Omni 等开源可自托 |
| 风险 | 误差级联 | 厂商锁定 |

- **2026 延迟预算共识**：语音→语音往返理想 800ms，可接受 600–1000ms。
- LiveKit Agents **两派都支持**（voice assistant primitive 可接 cascading，也可 drop-in OpenAI Realtime / Gemini Live）。我们当前是 cascading，未来可平滑升级。

## 四、端到端语音模型对比（重点：Qwen3-Omni）

| 模型 | 参量/架构 | 协议 | 端到端延迟 | 自托管 | 语言 | 声音克隆 | 备注 |
|---|---|---|---|---|---|---|---|
| **Qwen3-Omni** | 30B-A3B MoE（Thinker-Talker） | **Apache-2.0** | **211ms**（音频）/ 507ms（视频） | ✅ 完全免费 | 输入19语/输出10语 + 文本119语 | ✅ 3s | 36项基准22项SOTA，中文WER超越GPT-4o，可商用 |
| MiniMind-O | 115M / 312M-A（Thinker-Talker） | Apache-2.0 | 理论级低 | ✅ | 中/英 | ✅ | 极轻量，适合边缘 |
| Moshi (Kyutai) | ~7B（Helium） | CC-BY-4.0 | 理论160ms | ✅ | 仅英文 | 固定音色 | 端到端先驱 |
| GPT-4o-realtime | 闭源 | 专有 | <300ms | 否 | 多语 | 否 | 厂商锁定 |
| Gemini Live | 闭源 | 专有 | ~150ms TTS | 否 | 125+ | 否 | 厂商锁定 |

**Qwen3-Omni 与 HomeStream 品牌高度契合**：完全自托管、Apache-2.0 免费商用、211ms 端到端、中文优先、支持声音克隆——是 v5.4+ 「自托底语音」的天然候选，可替代当前「Ollama 文本 LLM + EdgeTTS 两段式」。

## 五、流式 TTS 与可观测性

### 5.1 流式 TTS 候选（替换 EdgeTTS 兜底）
| 方案 | 自托管 | 中文自然度 | 流式 | 备注 |
|---|---|---|---|---|
| **CosyVoice2** | ✅ | 高（含情感/韵律） | ✅ | 阿里的，模型已下 2GB 待接，契合自托管 |
| Fish-Speech | ✅ | 高 | ✅ | 开源音色克隆强 |
| Cartesia / Inworld TTS-2 | 否（API） | 极高 | ✅ | Realtime TTS 榜首，但非自托管 |
| ElevenLabs | 否 | 极高 | ✅ | 闭源商用 |

### 5.2 语音可观测性（给 v5.3.5 面板铺路）
- **Pipecat** 内建 OpenTelemetry 集成 + Whisker(实时调试)/Tail(监控)。
- **LiveKit** 为 BYO instrumentation（需自己接 OTel）。
- **关键指标**：TTFA P95 < 800ms、barge-in（打断）failure rate、turn-taking accuracy、音频质量/ASR 漂移。
- 建议：自研轻量 Trace（TTFA / 轮次 / 打断成功率）写入 v5.3.5 可观测面板语音子集。

## 六、对 HomeStream v5.2.5 的借鉴与增强路线建议

结合现状（EdgeTTS 兜底 + 已调优 VAD/LLM超时 + 待接 CosyVoice2 + Qwen3-Omni 候选），给出三条可并行/递进的增强路线：

| 路线 | 内容 | 收益 | 工作量 | 与品牌契合 |
|---|---|---|---|---|
| **A. CosyVoice2 流式中文 TTS** | 接已下 2GB 模型，替换 EdgeTTS | 中文自然度↑、情感韵律、声音克隆、**完全自托管** | 中（已备模型） | ⭐⭐⭐ 免费托底 |
| **B. Qwen3-Omni 端到端语音** | v5.4+ 候选，替代两段式，迈向 211ms | 跨越式低延迟、不降智、统一模型 | 大（需 vLLM/GPU 部署） | ⭐⭐⭐ 自托底语音终极形态 |
| **C. 语音 Trace 可观测性** | OTel/自研轻量指标（TTFA/打断率） | 量化后再优化、v5.3.5 铺垫 | 小 | ⭐⭐ 工程治理 |

**今天建议顺序**：先对 v5.2.5 现有链路做端到端测试（验证昨晚调优稳定性，这是遗留的「明日亲测」）→ 再落地路线 A（CosyVoice2，见效快、契合自托管）→ 路线 C 作为测试期顺带采集指标 → 路线 B 留待 v5.4+ 按 8 月地基规划推进。
